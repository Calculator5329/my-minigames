# Sand — Layer 1 Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Ship Layer 1 of the `sand` cartridge — a node-graph circuit editor with tick-based simulation, built from a single switch/transistor primitive, with 8–12 truth-table puzzles that auto-compile into reusable gate components, a sandbox mode, and the luminous silicon-wafer visual spec from §6 of the design doc.

**Architecture:** A new cartridge at `games/sand/` following the existing `manifest.js + game.js` pattern. The game is modularized inside `game.js` into `model / sim / compile / test / view / ui / levels` IIFE submodules. Persistent state flows through the existing `Storage.getGameData('sand')` API. Static level content ships as JSON under `games/sand/data/`. Pure simulation logic is also exposed for `node --test` harness so the sim layer is unit-testable without a browser.

**Tech Stack:** Vanilla JS (existing NDP engine), HTML5 Canvas 2D, DOM overlays, JSON level files, Node's built-in `node --test` for headless sim tests.

**Project reality check:** this working copy is **not a git repo**, so commit steps below are marked optional. If a repo is initialized later, re-enable them.

**Reference:** Design doc — [2026-04-19-sand-design.md](2026-04-19-sand-design.md).

## Task 6: Simulator — combinational stability (TDD)

**Files:**
- Create: `games/sand/lib/sim.js`
- Create: `games/sand/test/sim.test.mjs`

**Step 6.1** Failing test: hand-build a NOT out of one switch + power + ground + pad:

```
pad_in.out → switch.gate
power.out  → switch.in
switch.out → pad_out.in
```

Actually NOT is built by: `pad_in → switch.gate`, `power → switch.in`, `switch.out → pad_out` **with a pull-up on pad_out** — but our switch is pull-down by default, so the simplest Layer-1 NOT is:

```
pad_in → switchA.gate
power  → switchA.in
switchA.out → out          // this is BUFFER, not NOT
```

Correct NOT construction using only the n-type + power/ground: put the switch between `power` and `out`, and a pull-down resistor on `out`. In our abstraction, we model this by adding a second primitive `pullup` — a 1-tick-delay primitive that outputs 1 when its input is floating and 0 when driven low. **Update `primitives.js` to include `pullup`** (design doc update note: amend §3 primitive table to include `pullup`).

Revised NOT: `pad_in → switchA.gate; ground → switchA.in; pullup on switchA.out; switchA.out → pad_out.in` → output is 1 when gate is 0, 0 when gate is 1. That's NOT.

**Step 6.2** Write the test: run the sim for each input ∈ {0,1}, expect pad_out ∈ {1,0}.

**Step 6.3** Implement `Sim.create(graph) → runner`, `runner.setInput(label, value)`, `runner.step() → bool settled`, `runner.readOutput(label)`. Evaluation = dirty-flagged BFS from changed nodes; floating pins resolve to 0 unless a `pullup` pulls them high.

**Step 6.4** Add tests for AND (two switches in series between power and out with pullup), OR (two switches in parallel), NAND (series switches to ground with pullup), NOR, XOR (combination).

**Step 6.5** Implement until all gate constructions pass their truth tables.

**(Optional) Step 6.6** Commit: `feat(sand): combinational simulator + pullup primitive`.
